# Security Specification - Firestore Security Rules

This document outlines the security architecture, data invariants, and defensive validation strategies for the Firestore database of the Egyptian Universities Tansik Predictor.

## 1. Data Invariants

- **Collection: `community_questions`**
  1. **Authentication Requirement**: Any student creating a question MUST be authenticated (sign in anonymously or with Google).
  2. **Identity Integrity**: The `userId` field of the incoming question document must exactly match the authenticated user's ID (`request.auth.uid`).
  3. **Immutability of Author ID**: The `userId` field cannot be modified or spoofed once created.
  4. **Strict Schema Constraints**: No shadow fields are allowed. Only approved keys (`author`, `userId`, `track`, `score`, `question`, `answer`, `likes`, `createdAt`) are allowed on creation or update.
  5. **Timestamp Temporal Integrity**: The `createdAt` timestamp must be strictly validated against `request.time` (the server's timestamp) at creation and remain immutable.
  6. **Data Limits**: The question string must not exceed 1000 characters to prevent wallet-drain attacks and resource exhaustion. Name/Author must be <= 100 characters. Track must be <= 100 characters. Score must be <= 20 characters.
  7. **Answer Field**: The `answer` field is generated by advisors/AI, so standard students cannot modify the `answer` field directly or self-assign answers. Only specifically allowed operations (such as a separate role or action) can set this, or in our simplified setup, standard users can only create questions (with empty answer or default welcome message) or increment the `likes` field.

## 2. The "Dirty Dozen" Payloads (Unacceptable Writes)

The following payloads MUST return `PERMISSION_DENIED` under all conditions:

### Identity & Spoofing Attacks
1. **Unauthenticated Write**: Creating a question without a valid Firebase Auth credentials.
2. **Identity Spoofing**: User `studentA` attempts to create a question with `userId: "studentB"`.
3. **Identity Alteration**: Attempting to change `userId` from `"studentA"` to `"studentB"` on an existing question document.

### Schema & Injection Attacks
4. **Shadow Field Injection**: Creating a question with a secret field `isAdmin: true` or `flagged: false` not specified in the schema.
5. **Junk Fields / Denial of Wallet**: Creating a question with massive arbitrary keys (e.g., `tempPayload: "..."` over 1MB) to inflate storage.
6. **Volumetric Overflow**: Creating a question where the `question` field is 10,000 characters long (limit is 1000).
7. **Junk ID Poisoning**: Trying to create a document with an ID containing malicious symbols like paths or extremely long characters.

### State & Privilege Escalation
8. **Advisor Spoofing**: A regular student attempts to modify an existing question to write their own `answer` or edit a pre-written advisor response.
9. **Fake Timestamp Injection**: Attempting to set `createdAt` to a future or past date instead of `request.time`.
10. **Timestamp Alteration**: Attempting to update `createdAt` field on an existing question.
11. **Likes Manipulation**: Attempting to increase `likes` field directly by any arbitrary number (e.g. `likes: 99999`) instead of exactly `likes + 1` or via proper transactions.
12. **Foreign Delete**: A student attempting to delete a community question created by another user.
